# ゴルフ場情報スクレイピング機能 設計書

> **注記（2026-07-06追記）**: 本設計書の「共通マスタ（Com*）とユーザー専有登録（Usr*）を分離する」「pg_lake/pg_duckdbをアプリのクエリ経路に使う」という方針は、その後の議論で撤回された。最新の方針は以下の通り：
> - ゴルフ場マスタは単一（ユーザー専有登録は廃止）。一般ユーザーは閲覧・選択のみ、管理者ロールを持つユーザーのみが登録・編集できる（既存の `/golf-courses` 画面を管理者用画面として流用）
> - Icebergはスクレイピングした生データのステージング用途のみに使う。マスタ本体は通常のSupabase PostgreSQLテーブルとして管理し、Iceberg→Postgresへの取り込みはバッチ内でDuckDBを独立ツールとして実行し、結果を通常のPostgresクライアントでUpsertする（Postgres拡張としてのpg_duckdb/pg_lakeは使わない）
> - 詳細は [`docs/superpowers/plans/2026-07-06-golf-course-master-consolidation.md`](../plans/2026-07-06-golf-course-master-consolidation.md) 以降の実装計画を参照
>
> 「データモデル」章のCom/Usrプレフィックス、「保存方式」章のpg_duckdb/pg_lake関連の記述は、この注記をもって上書きされる。

## 背景・目的

ゴルフ場情報（ゴルフ場・コースレイアウト・ホールごとのPar/Yard）を都度手動登録するのはユーザーの負担が大きい。GDO・楽天GORA・じゃらんゴルフ・ShotNaviの4サイトから定期的にスクレイピングし、全ユーザーが利用できる共通マスタとして提供することで、手動登録の手間を削減する。

既存のゴルフ場登録機能（ユーザー専有登録）は維持し、共通マスタにまだ存在しないゴルフ場を補うための手段として引き続き利用できるようにする。

## スコープ

- 対象サイト: GDO、楽天GORA、じゃらんゴルフ、ShotNavi
- 中期: 自分・知人が実際にプレーする範囲（数百件程度）のゴルフ場をマスタ化
- 長期: 全国のゴルフ場（数千件規模）へ拡大。スクレイピング・バッチ構成は変えず、探索範囲（都道府県リスト）の設定変更のみで対応する

## データモデル

### プレフィックス命名規則

| 区分 | プレフィックス | 対象 |
|---|---|---|
| 共通マスタ（スクレイピングで生成、全ユーザー利用可） | `Com` | `ComGolfCourse` / `ComCourseLayout` / `ComHole` |
| ユーザー専有登録（既存機能） | `Usr` | `UsrGolfCourse`（旧 `MstGolfCourse`）/ `UsrCourseLayout`（旧 `MstCourseLayout`）/ `UsrHole`（旧 `MstHole`） |

既存の `Mst*` モデルは `Usr*` にリネームする（テーブル名・`@@map` 含む）。カラム名・外部キーの構造自体は変更しない。

### 共通マスタの位置づけ

- `ComGolfCourse` / `ComCourseLayout` / `ComHole` は既存の `UsrGolfCourse` 系と同じ形（ゴルフ場 → コースレイアウト（OUT/IN等） → ホール）を持つが、ホールのヤード情報はレギュラーティー1種類のみを保持する（既存 `Hole` モデルと同仕様）
- ユーザーに紐づかない（`userId` を持たない）
- アプリ層からの書き込み経路は存在しない。書き込みはスクレイピングバッチのみが行う

### ラウンド記録（`TrnRound`）のFK変更

- 既存の `golfCourseId` を `usrGolfCourseId`（nullable）にリネーム
- 新規に `comGolfCourseId`（nullable）を追加
- アプリ層でどちらか一方のみが値を持つことを保証する（DB制約（CHECK制約）で両方null/両方非nullを禁止する）
- `comGolfCourseId` にはPostgresの本物のFOREIGN KEY制約を張らない（後述のIceberg/pg_duckdb構成のため）。参照整合性はアプリ層でのID存在チェックのみで担保する

## 名寄せ・優先順位マージロジック

### 名寄せキー

ゴルフ場名＋都道府県・市区町村を正規化（表記ゆれ吸収: 「ゴルフ倶楽部」「GC」等の表記ゆれを吸収するルールを用意）した上で突き合わせる。郵便番号・緯度経度による名寄せは今回は行わない。

### サイト優先順位

GDO → 楽天GORA → じゃらんゴルフ → ShotNavi の固定順。

マージはコース単位で行う。同一ゴルフ場が複数サイトで見つかった場合、優先順位が最も高いサイトに情報（ゴルフ場名・コースレイアウト・ホールのPar/Yard）があれば、そのサイトの値を丸ごと採用する。最上位サイトに情報がなければ次点のサイトの値を丸ごと採用する（項目単位での混在は行わない）。

## スクレイピングバッチ

### 実行環境

- golf-webリポジトリ内にスクレイピング用コードを置き、GitHub Actionsの別ワークフロー（例: `.github/workflows/scrape-golf-courses.yml`）として定期実行する（cron）。既存の `deploy.yml` と同様、手動実行（`workflow_dispatch`）も可能にする
- 別リポジトリへの分離は行わない（スキーマすり合わせコスト・ジョブ管理テーブルの接続コストが見合わないため）

### 探索範囲の指定

設定ファイル（例: `config/scraping-target-prefectures.json`）または環境変数で都道府県リストを保持する。中期は対象範囲を絞り、長期でリストを拡張するだけで対応する。DBテーブル化は行わない（管理画面がないため過剰）。

### 再開可能なバッチ設計

1回の実行で全件処理せず、上限件数までを処理して中断・再開できるようにする。ジョブ進捗テーブル（例: `ScrapingJobProgress`：対象都道府県・サイト・処理済みカーソル位置を保持）をgolf-webのSupabase Postgres内に置く。

### 再スクレイピング周期

`lastScrapedAt` が3か月以上前のゴルフ場を優先的に再スクレイピング対象とする。

### 保存方式（要調査事項あり）

- スクレイピングで得た生データ（サイト別の抽出結果）と、名寄せ・優先順位マージ後の共通マスタデータは、AWS S3 または GCP Cloud StorageにApache Iceberg形式で保存する
- golf-db（BigQuery）側はこのIcebergテーブルを直接参照する（BigLake等）。BigQueryとの間に個別の連携・同期バッチは作らない
- golf-web（Supabase Postgres）側は `pg_duckdb` 拡張経由でこのIcebergテーブルをクエリする想定
  - **調査スパイク（実装着手前に必須）**：`pg_duckdb` がSupabaseのマネージド環境で有効化・利用可能かを検証する。利用できない場合は代替手段（他のFDW、または以下のフォールバック）を検討する
  - `pg_duckdb` は読み取り中心の想定のため、Icebergへの書き込みはスクレイピングバッチ側が担当し、golf-webのアプリ・Postgres側は参照のみ行う
  - `ComGolfCourse` 系のPostgres上の実体は、`pg_duckdb` 経由の外部参照になる可能性が高く、その場合Prismaの通常モデル管理外となるため、アプリコードからは生SQL（`$queryRaw`）でアクセスする
- **フォールバック**：`pg_duckdb` がSupabaseで利用できない場合、スクレイピングバッチが同一実行内でIcebergへの書き込みに加えて、Supabase Postgresのネイティブテーブル（`ComGolfCourse`等）にも書き込む。BigQuery向けのIceberg出力とPostgres向けの書き込みは同一バッチ内の並行出力であり、事後的な同期処理ではない

### 書き込み権限

共通マスタへの書き込みは、アプリのServer Action層には一切経路を設けない。スクレイピングバッチ専用のDB接続（service role）のみが書き込みを行う。

## UI・利用側の変更

### ゴルフ場選択

ラウンド記録のゴルフ場選択画面では、共通マスタとユーザー専有登録を同じ一覧に表示する。ユーザー専有登録の項目にのみ「ユーザー登録」ラベルを付けて区別する。

```
赤羽ゴルフ倶楽部
赤羽ゴルフ倶楽部  [ユーザー登録]
```

### ゴルフ場登録機能

`/golf-courses` 配下の既存機能は変更しない。引き続き、登録したユーザー本人のみが利用できる。

## 分析への影響

- `comGolfCourseId` を参照するラウンドは、ユーザー横断の平均値等の分析対象にできる（同一コース判定が共通マスタにより保証されるため）
- `usrGolfCourseId` を参照するラウンドは、ユーザー間で同一コースかどうかを判定できないため、分析対象外とする（既存動作を維持するのみで実現）

## テスト方針

- 各サイトのパーサーは、既存の `src/lib/parsers/gdoScorecard.ts` と同様、固定サンプルHTML/テキストを使ったユニットテストを追加する
- 名寄せロジック・優先順位マージロジックは、表記ゆれパターン（「ゴルフ倶楽部」「GC」等）を含むテストケースを用意する
- `Mst*` → `Usr*` のリネームに伴う既存機能（ゴルフ場登録・GDOインポート・ダッシュボード集計等）の回帰テストを実施する

## 未決事項・リスク

- `pg_duckdb` のSupabaseマネージド環境での利用可否は未検証。実装計画の最初のステップとして調査スパイクを行う
- スクレイピング元サイトのページ構造変更によりパーサーが壊れるリスクは常に存在する。バッチ内でのエラーハンドリング・失敗コース単位でのスキップ・ログ記録は実装計画側で詳細化する

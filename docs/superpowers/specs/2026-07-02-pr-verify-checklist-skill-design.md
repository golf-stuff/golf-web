# PR確認ポイント検証skill 設計

## 背景・目的

PR（例: [#2](https://github.com/golf-stuff/golf-stuff/pull/2)）には「確認ポイント」チェックリストが本文に含まれるが、これまでは人手で1つずつ動作確認していた。この作業をAI駆動で進め、Playwrightでブラウザ操作を行いながらユーザーと対話的に確認を消化できるようにする。

また、この手順は golf-web に限らず今後も使い回したいため、汎用的なskillとして切り出す。

## 成果物

1. `golf-web/CLAUDE.md` — golf-webの実態に即した開発ガイド（新規作成。既存ルートのCLAUDE.mdは別プロジェクト向けの記述で golf-web の実装と一致していないため、golf-web配下に実態に合わせたものを別途用意する）
2. `golf-web/.claude/skills/verify-pr-checklist/SKILL.md` — 汎用的な「PR確認ポイント検証」skill

## 全体アーキテクチャ

```
golf-web/
├── CLAUDE.md
└── .claude/
    └── skills/
        └── verify-pr-checklist/
            └── SKILL.md
```

- 起動: PR番号を指定して手動起動（例: `verify-pr-checklist 2`）
- 実行ステップ:
  1. `gh pr view <番号> --json body` でPR本文を取得し、`## 確認ポイント` 見出し配下の `- [ ]` 項目をパースする。見出しが見つからない場合はユーザーに確認ポイントを列挙してもらう
  2. 環境準備（`npm run dev` の起動確認、`supabase status` の確認。未起動ならskillが起動する）
  3. Playwright MCPでブラウザを操作し、各確認ポイントを検証（詳細は後述の分類ルール）
  4. 全項目終了後、結果をサマリーし、OK項目のみ `gh pr edit` でPR本文の `- [ ]` を `- [x]` に更新
  5. NG項目があれば詳細レポートを提示する（PR本文へは反映しない）

## ブラウザ操作基盤

Playwright MCP（`webapp-testing` skillが使う仕組み）を採用する。

- 採用理由: Claude Code のセッション専用機能である Claude Preview MCP と異なり、Playwright MCPはNode.jsから直接呼び出せるためCIやリモート環境でも同じ手順を再利用できる。skillの汎用性・将来のリモート実行ニーズと合致する

## 確認ポイントの分類ルール（ハイブリッド方式）

各確認ポイントの文言をAIが読み、以下のいずれかに分類する。迷った場合は主観系（ユーザー主導）に倒す。

- **定型・機能系**（例:「〜が動作する」「〜がある」「〜を押すと〜になる」）
  → AIがPlaywrightで実際に操作（フォーム入力・クリック・保存など）し、DOM状態やコンソールエラーの有無を検証する。結果（操作ログ・スクリーンショット）を提示し、ユーザーには「この結果で問題なければ次へ」という軽い最終確認のみ求める

- **主観・見た目系**（例:「デザインが統一されている」「レイアウトが崩れていない」）
  → AIは対象ページを巡回してスクリーンショットを撮るところまでは自動化するが、「統一されているか」の判断はユーザーに委ねる。撮影したスクリーンショットを提示するか、実機での確認を案内する

## 環境準備

skillが以下を自動で行う。ユーザーはPR番号を渡すだけでよい。

- `npm run dev` が起動していなければ起動する
- `supabase status` でローカルSupabaseが起動していなければ `supabase start` を実行する

## NG時の挙動

1件NGが出ても検証フローは中断せず、詳細（理由・再現手順）をその場で記録し、残りの確認ポイントの検証を継続する。全項目の検証が終わった時点で、NG項目をまとめて最終レポートに含める。

## PR本文更新 & 最終レポート

- 全項目検証後、OKだった項目のみ `gh pr edit <番号> --body "<更新後の本文>"` で `- [x]` に変更する。NG項目は `- [ ]` のまま残す
- チャット上の最終レポートは以下の形式とする:

```
## PR #2 確認結果

✅ /rounds/import でスコアインポートが動作する
   - 操作: コース選択→貼付→パース→インポート実行
   - 結果: 正常に保存、redirect確認

⚠️ 全ページのデザインが統一されている（ユーザー確認）
   - スクリーンショット: rounds一覧/詳細/ゴルフ場一覧を撮影済み
   - ユーザー確認: OK

❌ /rounds/[id]/holes に「GDOで上書き」ボタンがある
   - 詳細: ボタンは存在するが押下後に404
```

## ファイル内容の具体構成

### `golf-web/CLAUDE.md`

- プロジェクト概要（golf-webはラウンドのスコア記録・GDOインポート・ゴルフ場管理を扱うNext.js + Prisma + Supabaseアプリであること）
- ディレクトリ構成（`app/`, `src/lib/parsers/`, `prisma/`, `supabase/`）
- 開発コマンド（`npm run dev`, `npm run build`, `npm run test`, `npm run lint`）
- ローカルSupabase起動手順（`supabase start`、`.env.local`の位置）
- 「PRレビュー時は `verify-pr-checklist` skillでPlaywright検証を回せる」という導線の一文

### `golf-web/.claude/skills/verify-pr-checklist/SKILL.md`

- frontmatter: `name: verify-pr-checklist`, `description`（GitHub PRのチェックリスト形式の確認ポイントをPlaywrightで検証し、ユーザーと対話しながらチェックを消化するときに使う旨）
- 手順本体:
  1. PR本文取得・確認ポイントのパース手順（正規表現例、`## 確認ポイント` 見出しが無い場合のフォールバック）
  2. 環境準備手順（dev server / Supabase起動確認コマンド）
  3. 確認ポイントの分類ルール（定型/主観の判定基準と具体例）
  4. Playwright MCPでの検証手順テンプレート（ページ遷移→操作→アサーション→スクリーンショット）
  5. ユーザー確認の聞き方（定型系は軽い確認、主観系はスクリーンショット提示＋判断依頼）
  6. 結果集計とPR本文更新（`gh pr edit`）の手順
  7. 最終レポートのフォーマット

汎用性を持たせるため、golf-web固有の記述（PR #2の3項目など）はSKILL.md本体には書かず、例示程度に留める。

## スコープ外

- CI上での自動実行（今回は手動起動のみ）
- 確認ポイントの粒度自動分割（PR本文の記述をそのまま1項目として扱う）
